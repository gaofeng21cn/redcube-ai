# 典型交付示例

这份文档展示 `RedCube AI` 当前两类最重要的交付面：`PPT` 与 `小红书图文`。

## 示例一：给学生讲课的 PPT

### 适用场景

- 教授给本科生或研究生授课
- 专家做入门性培训
- 需要按讲述节奏逐页组织内容

### 典型输入

- 课程提纲
- 讲义笔记
- 论文或参考资料
- 必须强调的关键概念

### 期望输出

- 有清楚章节节奏的 `PPT deck`
- 每页承担明确讲述任务
- 结构、视觉和导出要求可审阅

### 示例指令

> 使用 RedCube AI 把这些材料组织成一份面向本科生的 PPT 演示文稿。目标是讲清甲状腺基础知识，要求结构清楚、节奏稳定、适合正式授课，并保留后续 review 与 rerun 的空间。

## 示例二：知识传播型小红书图文

### 适用场景

- 专家做面向公众的知识传播
- 希望把复杂内容转成更易理解的图文表达
- 需要兼顾信息密度与传播性

### 典型输入

- 原始专业材料
- 过往文章或草稿
- 平台风格要求
- 需要保留的专业边界

### 期望输出

- 可连续阅读的小红书图文结构
- 明确的页面节奏与关键信息层级
- 可继续审阅和重跑的交付过程

### 示例指令

> 使用 RedCube AI 把这些材料组织成适合发布的小红书图文。目标是向普通读者讲清甲状腺门诊常见问题，既要易懂，也要避免失真，并保留后续 review 与 rerun 的空间。

## 两类交付的共同点

- 都不是一次性“生成完就结束”
- 都依赖结构化阶段、审阅节点和导出约束
- 都适合由 Agent 推进、人类审核
- 都共享 `Gateway + Overlay + Runtime` 主线

## 两类交付的关键差异

| 维度 | PPT | 小红书图文 |
| --- | --- | --- |
| 首要目标 | 讲授、汇报、演示 | 传播、发布、快速理解 |
| 节奏单位 | 页 / 章节 | 页面 / 图文段落 |
| 重点约束 | 讲述顺序、版式结构、演示节奏 | 信息密度、传播结构、发布导向 |
| 适合的审阅问题 | 是否讲清了 | 是否传达准了、够不够易懂 |
