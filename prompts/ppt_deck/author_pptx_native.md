# author_pptx_native

Use the approved slide blueprint and visual direction to materialize an editable PowerPoint-native deck. Preserve the storyline, slide count, audience-facing text, evidence labels, and visual direction. The output contract is a `.pptx` with editable native text boxes and shapes plus a shape manifest for review.

Visible text must be project-facing. Do not copy operator constraints, local filenames, route names, prompt names, `RCA`, `RedCube`, `product-entry`, `source intake`, or “do not expose” instructions into editable slide text. If upstream title, goal, blueprint, or visual direction already contains internal names, rewrite them as audience-facing labels such as “自主交付链路”, “可复核交付闭环”, or “从目标到导出的验收路径”.

Keep the title safe zone clear of section chips, left-top cards, badges, tags, and decorative labels. For tables, use at least 11pt body text, compact cell padding, and a dense readable grid rather than small text inside oversized empty cells.

Native PPTX authoring must pass the same readability floor as the rendered screenshot review:

- Treat the upstream `visual_direction` as source material, then emit a top-level `editable_shape_plan.design_spec_lock` before any slide shapes. The lock must include `spec_id`, `owner: "llm_agent"`, palette, typography, grid, motif, layout rhythm, at least three concrete `layout_archetypes`, borrowed design principles, forbidden motifs, and QA gates. Do not put this only under `design_system`; the materializer will fail closed if `design_spec_lock` is absent or thin.
- In `design_spec_lock.borrowed_principles`, include the external PPT-agent discipline RCA is adopting: `ppt_master_style_spec_lock`, `template_layout_grammar`, `template_profile`, `semantic_layout_selection`, `reference_deck_analysis`, `per_page_visual_plan`, `layout_rhythm`, and `rendered_quality_gate`. In `design_spec_lock.qa_gates`, include at least `bounds`, `font_floor`, `text_fit`, `structural_visual`, and `layout_variety`.
- Include `design_spec_lock.professional_design_brief` before coordinates. It must name the design register, reference-style family, first-glance hierarchy, template profile strategy, capacity strategy, and forbidden amateur patterns. If upstream `visual_direction` is generic, synthesize a professional design brief first; do not materialize generic cards and expect QA to rescue the slide.
- Emit `editable_shape_plan.deck_layout_rhythm_plan` before slide shapes. Set `owner: "llm_agent"` and `required: true`. This is the contact-sheet plan: one row per slide in `slides[]` with `slide_id`, `rhetorical_role`, `selected_archetype`, `primary_grid`, `composition_signature_budget`, and `proof_object`. Do not use the same `selected_archetype` or `primary_grid` three slides in a row, and keep at least 75% of `composition_signature` values distinct in normal decks. This is a hard design prerequisite, not a review note.
- Before emitting any `native_shapes`, emit a top-level `editable_shape_plan.template_layout_grammar`. This is the professional PPT design skeleton, separate from style. It must include `owner: "llm_agent"`, `required: true`, `materializer_role: "execute_selected_archetype_zones_only"`, `helper_template_layout_allowed: false`, global rules for edge margin / zone gap / zone inset / connector lanes, and an archetype catalog with professional layouts such as `professional_system_map`, `executive_status_board`, `evidence_timeline`, `risk_control_matrix`, and `decision_dashboard`.
- Borrow the useful pattern from ppt-master, PPTAgent, officecli-pptx, Presenton, and ppt-agent-skills before coordinates: treat templates or reference decks as layout intelligence. Even when the user provides no template file, include a `template_layout_grammar.reference_discipline` object with `template_profile_required: true`, `semantic_layout_selection_required: true`, `placeholder_capacity_required: true`, `reference_deck_analysis_required: true`, `action_title_required: true`, and `source_projects: ["ppt-master","PPTAgent","officecli-pptx","presenton","ppt-agent-skills"]`. The selected archetype must behave like a semantic master layout with zones, placeholder capacity, spacing rhythm, and action-title hierarchy.
- Each `template_layout_grammar.archetype_catalog[]` entry must be a real layout contract, not just an id. Include `use_when`, `layout_description`, `required_zones`, `content_schema.required_shape_roles`, and `prohibited`. Borrow the discipline of template-aware PPT agents: the layout contract says which semantic slots exist and which mistakes are forbidden before any coordinate is emitted.
- Treat templates as layout intelligence, not visual skin. If no user PPT template is supplied, create an internal master-layout equivalent in `template_layout_grammar`: named archetypes, semantic placeholders, placeholder capacity, zone hierarchy, spacing rhythm, and prohibited fit failures. Do not call a palette, background, or repeated card grid a template.
- Validate capacity before coordinates: the selected archetype must have enough zone area for the actual text at the typography floor. If it does not fit, reduce slots, shorten copy, or select another archetype before emitting shapes.
- Each archetype `content_schema` must also declare `required_shape_role_groups` and `min_filled_required_zone_share`. The selected archetype is accepted only if the actual shapes fulfill those role groups and fill enough required zones; declaring `decision_dashboard` or `executive_status_board` while drawing generic cards is a hard preflight failure.
- Every zone and every shape must use canonical `bounds: { "left_in": number, "top_in": number, "width_in": number, "height_in": number }`. Never use aliases such as `x`, `y`, `w`, `h`, `left`, `top`, `right`, `bottom`, `width`, or `height`; alias bounds fail before Python/officecli materialization.
- Every slide must include `template_layout_binding` before `native_shapes`. The binding must select one archetype from the grammar, set `archetype_instance_id`, set `rhythm_role`, set `zone_gap_in_min >= 0.32`, set `zone_inset_in_min >= 0.15`, and declare semantic zones such as `title_zone`, `claim_zone`, `system_map_zone`, `gate_zone`, `evidence_zone`, `timeline_zone`, `matrix_zone`, `decision_zone`, `proof_zone`, or `takeaway_zone`. Each zone needs `zone_id`, `semantic_role`, `bounds`, `intended_content`, `min_font_pt`, and `safe_inset_in >= 0.15`.
- Every non-decorative/non-auxiliary shape must carry `layout_zone_id` pointing to one declared zone on the same slide, and the full shape bounds must stay inside that zone. Page numbers, footers, dates, source notes, speaker identity labels, cover meta, and purely decorative accents may omit it only when they use `quality_role: "auxiliary"` or `"decorative"`; structural rails, connectors, proof bands, evidence axes, decision rails, gate panels, content panels, labels, route cards, gate cards, evidence notes, tables, charts, metric grids, and audience text boxes must bind to a zone. This prevents the AI from inventing unplanned floating cards or floating structural lines after the layout is supposedly locked.
- Use the design discipline of strong native PPT builders: claim spine first, design spec lock second, contact-sheet rhythm third, explicit editable shape coordinates fourth, rendered QA last. RCA owns this workflow; ppt-master, officecli-pptx, PptxGenJS, Marp, Slidev, Presenton, and similar projects are references for discipline only, not the workflow owner.
- Every slide must include `layout_intent` with `rhetorical_role`, `composition_signature`, `primary_grid`, `visual_weight`, `negative_space_strategy`, `non_text_visual`, and `forbidden_template_reuse_checked: true`. The `composition_signature` must describe concrete geometry, not just a layout family label.
- In `layout_intent.non_text_visual`, name the structural object, not just “cards” or “panels”. Good examples: `status board with gate stack, evidence band, metric markers, and separated connector lanes`; `decision dashboard with proof band and gate stack`; `timeline rail with milestone nodes above labels`. Generic phrases such as `three editable cards` or `editable status cards` will fail unless the shape plan also contains explicit structural roles.
- Every `native_shapes[]` item must include `shape_id`, `kind`, `role`, `quality_role`, `bounds`, and either `editable_text` for text-bearing shapes or visible fill/line styling for purely non-text shapes. Allowed `quality_role` values are `content`, `structural`, `decorative`, and `auxiliary`.
- Do not rely on materializer defaults for design. Every visible text-bearing shape must include an explicit `font_size`; every non-text content or structural shape must include explicit visible `fill` or `line`; every visible shape must include an explicit valid `quality_role`. `slide_blueprint.slides` is context only and cannot replace `editable_shape_plan.slides`.
- For `native_visual_sample: true` or any one-slide sample, design a low-density executive slide instead of a dense proof ledger. Use no more than one title, one core sentence, three route/gate items, one takeaway, and one compact evidence note. Keep total text-bearing shapes at 12 or fewer and avoid multi-row receipt lists.
- For `native_visual_sample: true`, follow `native_ppt_sample_layout_profile` when present. Use only `sample_status_proof_board` or `sample_decision_proof_split` as the selected archetype and include that archetype in `template_layout_grammar.archetype_catalog`. Do not select `executive_status_board`, `decision_dashboard`, `professional_system_map`, `evidence_timeline`, or `risk_control_matrix` for a one-slide sample; those are general deck archetypes and are too loose for this proof.
- For `native_visual_sample: true`, use the external PPT-agent discipline as a generation-time design gate: lock a capacity-safe sample page before coordinates. Exactly one of these two layouts is allowed: (a) `sample_status_proof_board`: title + claim + one visible `input_hub` feeding three large status cards + short `flow_connector`/`merge_connector` shapes into one full-width proof band, with no separate takeaway panel; merge the takeaway into the claim or proof sentence. (b) `sample_decision_proof_split`: title + claim + left decision panel + right proof stack + visible `decision_rail` / `proof_rail` + one bottom takeaway band. Do not combine status cards, evidence band, evidence axis, and a separate takeaway panel on the same one-slide sample.
- For `native_visual_sample: true`, the evidence/proof text must be one compact sentence, not an artifact inventory. Keep it under 28 CJK characters or 18 English words when possible, for example: `证据同时覆盖文件、截图和审阅记录。`. Do not list PPTX, PDF, screenshot, shape manifest, review receipt, and export receipt as separate clauses inside the slide; those belong in artifact metadata, not the visible one-slide board.
- For `native_visual_sample: true`, the page still needs a visible editable structural visual. Add at least two non-text support shapes, including at least one `quality_role: "structural"` object such as a proof band, decision rail, input hub, evidence axis, gate stack panel, or connector that links the input to the proof/decision. Do not rely on content cards themselves as the structural visual. The visual director must be able to read a causal path in the screenshot: shared input -> three route outputs -> closed proof/decision.
- For `native_visual_sample: true`, a shape marked `quality_role: "structural"` must have a structural role name such as `proof_band`, `decision_rail`, `input_hub`, `evidence_axis`, `gate_stack_panel`, `flow_connector`, `bridge_connector`, or `metric_grid`. A generic `content_panel` with `quality_role: "structural"` is not a structural visual; rename the role and make it carry the first-glance logic, or keep it as content.
- A one-slide executive board must not be only five `content_panel` cards plus text. If the plan has multiple equal cards, add a dominant proof/decision zone and a structural band/rail/axis that creates first-glance hierarchy, or reduce/merge cards. A generic equal-card grid will be rejected as `ai_first_mechanical_card_template_detected`.
- Structural visuals are planned shapes, not free-floating decoration. Every `quality_role: "structural"` rail, connector, band, axis, hub, bridge, or gate panel must set `layout_zone_id` to a declared zone such as `status_zone`, `proof_zone`, `evidence_zone`, `decision_zone`, or `takeaway_zone`.
- For `native_visual_sample: true`, every selected zone must be large enough for its text before any shape is emitted. For each content shape, check `left/top/right/bottom` against the declared `template_layout_binding.zones[]` bounds and keep at least 0.02in zone tolerance plus any panel safe inset. If the shape cannot fit its declared zone, redesign the zone geometry or reduce slots; do not return a shape that hangs outside the zone.
- For `sample_status_proof_board`, declare only `title_zone`, `claim_zone`, `status_zone`, and `proof_zone`. Do not declare `takeaway_zone`. Reserve at least 1.15in for title, 1.05in for claim, 3.45in for status board, and 1.35in for proof band. Include a dominant centered `input_hub` at least 10.4in wide and 0.82in high, with 22pt visible hub text `同一材料同步进入三条路线验证`, spanning all three card centers. Draw exactly three straight vertical `kind: "connector"` arrows from hub bottom to the card top centers; each drop should be about 0.03-0.04in wide, at least 0.66in high, directional, and clear of text. Do not draw a horizontal bus.
- For `sample_status_proof_board`, use exactly three status cards: three `content_panel` shapes and three paired `point_text` shapes in `status_zone`. Each card must be at least 4.0in wide and 1.35in high. Each paired `point_text` box must sit fully inside its card with at least 0.15in safe inset on all sides. Do not compress card height below the text floor to make room for extra bottom notes.
- For `sample_status_proof_board`, mark the three `content_panel` card backgrounds as `quality_role: "content"`. Do not mark card backgrounds as `quality_role: "structural"`; the structural objects are the input hub, flow/merge connectors, and proof band.
- For `sample_status_proof_board`, the `input_hub` is a major flow node, not a small label. Use `同一材料同步进入三条路线验证`, center it across the route row, and make the shape visually as wide as the three-card group. Do not shrink it to a left-card label or replace the three drops with isolated ticks, a bracket, or a horizontal bus.
- For `sample_status_proof_board`, `layout_intent.non_text_visual` must spell out the first-glance path: shared input -> three route/status cards -> proof band. A generic phrase such as `input_hub with flow_connector lanes` is not enough.
- For `sample_status_proof_board`, every `flow_connector` or `merge_connector` must be fully inside its declared `layout_zone_id` safe bounds. Keep split connectors inside `status_zone`. If you bind a merge connector to `proof_zone`, place it inside the top of `proof_zone`; do not draw it in the gutter between `status_zone` and `proof_zone`. If the gutter is visually needed, enlarge one zone so the connector lane is claimed by that zone.
- For `sample_status_proof_board`, `proof_zone` may contain only one audience-facing proof/evidence sentence. Do not add `boundary_note`, scope text, a second evidence text box, or visible artifact inventory in the proof band. Put scope and boundary details in artifact metadata, speaker notes, or review receipts, not on the slide.
- For `sample_decision_proof_split`, reserve at least 1.15in for title, 1.05in for claim, 3.00in each for decision/proof middle zones, and 1.20in for the bottom takeaway zone. Include a visible `decision_rail` or `proof_rail` connecting the decision panel to the proof stack and takeaway band. If the text does not fit, shorten or merge content before returning coordinates.
- For `native_visual_sample: true`, return a complete `template_layout_binding` on every retry. Preserve `template_layout_binding`, `layout_intent`, `native_shapes[].layout_zone_id`, and the top-level `template_layout_grammar` exactly unless a structural failure explicitly names them. A geometry or text-fit retry that drops `template_layout_binding` is invalid even if the visible coordinates improve.
- In one-page samples, make each route card a large `content_panel` or `rounded_rect` of at least 3.3in wide by 1.55in high, and place its paired `route_label` text box inside with `left_in >= panel.left_in + 0.22`, `top_in >= panel.top_in + 0.22`, `width_in <= panel.width_in - 0.44`, and `height_in <= panel.height_in - 0.44`. Use `font_size: 18`, `height_in >= 1.05`, and one complete short sentence. If this cannot fit, shorten the sentence or use two route cards, not smaller text.
- For one-page samples, avoid `role: "route_label"` when the text is sentence-length. Use `role: "point_text"` inside status cards for complete sentences, or keep `route_label` to 8 CJK characters / 16 Latin characters or fewer with `width_in >= 4.2`. Three sentence-length route labels in 2.8in boxes will fail.
- Do not put `evidence_item`, `metric`, and `takeaway` text in the same background panel. Give each text box its own container or use one single evidence sentence. If you draw an evidence band, place only one text box inside it with at least 0.15in inset.
- Use `quality_role: "content"` for audience-facing slide text and content panels. Use `quality_role: "structural"` for first-glance visual structure such as rails, connectors, axes, timeline tracks, hubs, bands, comparison bridges, metric grids, charts, and tables. Use `quality_role: "decorative"` only for low-information accents. Use `quality_role: "auxiliary"` for page numbers, footer, source note, date, speaker identity label, or cover meta.
- Structural visuals must be real editable shapes with materializable geometry, not just role names on cards. Prefer `line`, `connector`, `oval`, `rect`, `rounded_rect`, `table`, `chart`, or `metric_grid` roles containing `rail`, `connector`, `axis`, `timeline`, `flow`, `hub`, `band`, `bridge`, `metric`, or `map`.
- Use only officecli-materializable `kind` values: `text_box`, `text`, `rect`, `rectangle`, `rounded_rect`, `panel`, `oval`, `circle`, `line`, or `connector`. Do not emit `triangle`, `right_triangle`, `polygon`, `arrow`, `icon`, `group`, `freeform`, `image`, `svg`, `path`, or custom shape kinds. Build icons from small `rect`, `line`, and `oval` primitives instead.
- Page numbers, speaker identity labels, and cover metadata must use roles such as `page_number`, `page_no`, `speaker_identity`, `cover_meta`, or `footer` with `quality_role: "auxiliary"`. Do not mark auxiliary metadata as body content.
- Keep at least 75% of slide `composition_signature` values distinct in a normal deck, and never repeat the same concrete composition on three slides. Change the first-glance structure: hero callout, split comparison, horizontal timeline, gate ladder, radial/system map, synthesis band, chart/table/metric grid, or connector flow.
- Layout variety is checked before materialization. A deck that merely changes text while reusing the same card grid, selected archetype, or primary grid three times in a row will be rejected before officecli writes the file.
- Use PPT design floors from officecli-pptx as hard constraints: title at least 36pt, body at least 18pt, step/index labels at least 16pt, table body at least 11pt, 16:9 edge margin about 0.6in or more, and visible gaps between blocks. Solve fit by changing geometry, shortening titles, or reducing slots, not by shrinking text. For CJK titles at 40-48pt, reserve at least 1.55in title height when the title can wrap, or widen the title box enough that it stays one line.
- Keep title-to-body hierarchy at 2.0x or stronger. If the largest body/core text is 20pt, use a 40pt or larger title; if the title must stay 36-38pt, keep body/core text at 18pt. A 38pt title over 20pt body is visually flat and will fail `typography_hierarchy_ok`.
- In one-slide native samples, target a 40-44pt title and keep body/core/point/proof text at 18pt unless the title is larger. If a 44pt CJK title would wrap, shorten the title or give it a full-width, 1.55in-high box; do not flatten hierarchy by pairing a 36-38pt title with 20-21pt body text.
- Keep all audience-facing content at least 0.60in from every canvas edge. Bottom evidence bands, boundary notes, and page numbers are common failure points: the lowest content text box must end no lower than 8.05in on a 16:9 slide, with the auxiliary page number below it or outside content QA.
- Title and core sentence boxes must have visible text separation, not just geometric boxes that nearly touch. Put the core sentence at least 0.18in below the estimated title text bottom; for 44pt CJK titles this usually means either title height >= 1.30in with a one-line title, or core top >= 1.70in when the title starts near 0.46in.
- Size every text box from the actual text before emitting coordinates. At 18pt, a one-line CJK/Latin body label needs at least 0.36in usable height after margins; reserve 0.58in or more for compact labels and 0.90in or more for sentence-length body text. For 20-24pt lead, intro, thesis, takeaway, or core sentence boxes, reserve at least 0.95in height for sentence-length text; 0.72in is too short and will fail preflight even if it looks like a one-line caption. Do not place 18pt `point_text` in 0.3-0.4in high boxes.
- Treat text margins and visual container padding as part of the design, not as export trivia. Audience-facing content text boxes use a 0.10in default internal margin, and any text centered inside a `content_panel`, `input_panel`, or `source_panel` must remain fully inside that panel with at least 0.15in safe inset on every side. If the text touches or visually bleeds outside the card, enlarge the panel, shorten the copy, or move the text before returning the plan.
- Safe inset is a coordinate rule: if a text box is visually inside a panel, its left/top/right/bottom edges must all be at least 0.15in inside that panel. Do not let route labels, evidence notes, or metrics share the same coordinates as their panel background. A background panel and its text need different bounds.
- Short route and delivery-gate labels must read as balanced labels, not forced wrapped captions. For `route_label` or compact `gate_card` text of about 22 meaningful characters or fewer, make the text box wide enough for one line, or shorten the text. Do not allow labels such as route names or gate outcomes to split across two awkward lines in the middle of a system map.
- Leave explicit vertical and horizontal gaps between all text-bearing shapes. Panel titles, route labels, evidence metrics, badges, and explanatory body text may live in the same visual region, but their boxes must not overlap; use separate rows, columns, or a clearly stacked evidence band. Do not place a bottom evidence title, metrics, and body sentence in the same y-range.
- In system-map or workflow-map slides, route labels and metric badges must not sit directly on connector lines. Put any `metric`, `route_label`, or `evidence_item` badge above or below the line with at least 0.12in clearance, or route the connector around the badge. A label centered over a connector is a visual collision even when the text boxes do not overlap.
- Treat connectors, rails, tracks, and ladders as visible foreground geometry during planning. Before returning the plan, inspect each connector against nearby `route_label`, `point_text`, `evidence_item`, `gate_card`, `metric`, and `takeaway` boxes; if a line crosses readable text or sits inside its text box, move the line into an empty gutter or move the label away. The preflight rejects this as `ai_first_structural_text_collision`.
- Use gate area roles that QA can recognize. For the right-side delivery-gate zone, use `role: "gate_stack_panel"`, `role: "gate_ladder_panel"`, or `role: "gate_card"` on the structural container, and keep the gate markers and labels inside that area.
- Treat `native_shape_plan_validation_feedback.required_shape_fixes[]` as a hard patch list. For every listed `shape_id`, update that exact shape so `bounds.height_in >= required_height_in`; if there is no room, reduce slots or move neighboring shapes. Never return the same height for a listed failure.
- The validation feedback can require text and font fixes, not just height fixes. If `required_font_size` is present, set that same shape's `font_size` to at least that value. If `required_text_char_count` is present, rewrite that same shape's `editable_text` into a complete audience-facing phrase or sentence with at least that many meaningful characters after punctuation is removed.
- The validation feedback can require explicit `quality_role` fixes. If `required_quality_role` is present, set that exact shape's `quality_role` to that value. Allowed values are only `content`, `structural`, `decorative`, and `auxiliary`; never use `primary`, `secondary`, `visual`, `body`, `title`, or role-like labels as `quality_role`.
- The validation feedback can require geometry fixes for pairs of shapes. If `other_shape_id`, `required_gap_in`, or `geometry_repair_instruction` is present, move or resize `shape_id` and/or `other_shape_id` so their visible text rectangles no longer overlap and keep the requested clearance. Do not rename the shape, hide the text, or mark it auxiliary to escape the failure.
- The validation feedback can require exact layout-zone containment. If `required_inside_zone` and `zone_bounds` are present, repair the exact `shape_id` so its whole bounds fit inside the declared `layout_zone_id` zone; if there is not enough usable room, expand/redesign the zone or reduce the number of slots instead of leaving the shape outside the AI-selected template. Treat `zone_bounds` as a hard geometry contract, not a visual suggestion.
- The validation feedback can require a minimum label width. If `required_width_in` is present, set the same text box to that width or shorten/re-role the label so it does not wrap awkwardly. Short gate labels should be one-line labels; sentence-length explanations should become `point_text` inside a larger content panel.
- The validation feedback can require a structural redesign, not just exact shape resizing. If the failure mentions `ai_first_visual_structure_missing`, `ai_first_visual_support_shape_count_too_low`, or `ai_first_mechanical_card_template_detected`, rebuild the slide skeleton around a real editable structural visual: proof band, decision rail, input hub, evidence axis, gate connector, or flow connector. Update `layout_intent.composition_signature`, `layout_intent.non_text_visual`, `template_layout_binding.zones`, and `native_shapes` together. Do not return the same equal card grid with one more decorative shape.
- If structural retry feedback names `native_shapes[...].layout_zone_id`, repair that exact structural shape by assigning it to a declared zone and keeping it inside that zone. Do not delete the structural shape or mark it decorative/auxiliary to bypass the contract.
- The validation feedback can include `global_shape_class_fixes[]`. These are not suggestions: apply them to every matching role/class on the slide, even if only one shape failed last time. If any 18pt `metric`, `evidence_item`, `boundary_note`, `point_text`, `route_label`, or `takeaway` failed height or overlap, raise all similar content text boxes to at least 1.05in high, keep width at least 2.8in, and reduce the number of bottom evidence/metric blocks instead of moving the problem to the next label.
- For short `panel_title` labels at 18-22pt, reserve at least 0.68in height by default. For any `point_text` or `body` sentence expected to wrap, reserve at least 1.25in height, or make the text box wide enough that the validator estimates fewer lines. Do not use 0.78in sentence text boxes.
- For one-slide system maps, route and evidence descriptions should be compact but complete: one short sentence per route, 18pt, width >= 2.8in, height >= 1.05in; if the text needs more than two lines, shorten the sentence or make the panel wider. Bottom evidence notes must not share vertical bands with metrics or takeaways.
- Every `line` or `connector` shape must have positive materializable thickness in `bounds`: horizontal rules use `height_in >= 0.03`, vertical rules use `width_in >= 0.03`, and diagonal/bridge connectors use both positive width and height. Never emit zero width or zero height, even for footer rules.
- Short labels are still text boxes. `speaker_identity`, `route_label`, compact badges, panel headings, and boundary notes need at least 0.64in height at 18pt unless the validator asks for more. Do not use 0.38-0.45in text boxes for visible labels.
- For every failed `line` or `connector` in `native_shape_plan_validation_feedback.required_shape_fixes[]`, change the same shape so both `bounds.width_in` and `bounds.height_in` are positive; the thin dimension must be at least 0.03in. A vertical gate ladder should be width 0.03-0.06in, never 0.
- Use ppt-master-style spec discipline: lock the design system before page shapes, keep editable shapes grouped by semantic purpose, emit a manifest-friendly plan, and expect screenshot/render QA to fail closed on drift.
- Do not draw the common AI-slide title underline motif. Use whitespace, a side rail, a top band, a corner anchor, connector flow, or page-specific visual object instead.
- Every content slot that is visually drawn must be filled with audience-facing text; never emit a four-card or four-quadrant shape plan with only two or three real content slots.
- If a visual panel is a background/container only, mark it as `role: "content_panel"` with no text and pair it with a nearby filled `point_text` shape. Do not expect the materializer to infer slot text from an empty panel alone.
- Each `point_text` must be a complete short teaching sentence with at least 12 meaningful CJK/Latin characters after punctuation is removed. Labels such as “目标：这页要教会什么” are too thin unless they explain the action and evidence.
- Short badge or receipt labels such as files, screenshots, manifests, and export receipts should use `role: "evidence_item"` or `role: "metric"` rather than `point_text`; they are allowed to be compact labels, but they still need readable font size and sufficient box height.
- Do not create six separate receipt/evidence labels in a one-page native sample. Summarize proof artifacts in one readable evidence note such as “文件、PDF、截图、形状清单和审查记录同时存在”，or use three metric badges at most. More tiny receipts create 16pt clutter and will fail native preflight.
- Delivery-gate labels are audience content, not decorative captions. Do not write only “视觉审阅”, “截图审阅”, or “演示导出” as `gate_card`, `route_label`, or `point_text`; write complete phrases such as “视觉审查确认页面结构可读”, “截图审查确认问题可以复核”, or “演示文稿导出确认文件可交付”.
- Route or stage labels such as `source readiness`, `storyline`, `detailed outline`, `slide blueprint`, `visual direction`, `screenshot review`, and `export` are never sufficient as standalone `point_text`; if needed, put them inside a sentence that explains what happens and what evidence is produced.
- `point_index` / step labels are visible audience-facing text, even when drawn inside an oval, circle, badge, or marker. Each `point_index` shape must include `editable_text` such as `"01"`, `"02"`, or `"03"`, a readable `font_size` of 16pt or larger, and enough marker bounds for the text. Do not use `shape_id`, role, fill color, or nearby body text as a substitute for the visible index label.
- Avoid three consecutive content slides with the same concrete layout variant or same `composition_signature`. Change the visual structure, not just the wording, when adjacent pages carry different rhetorical jobs.
- Prefer fewer, larger, well-filled cards over many tiny placeholders. If the content only has two or three points, choose a two-column or three-card composition rather than leaving an empty fourth slot.
